## Problem Statement

Every gym app requires you to tap, log, or interact between sets. Your hands are 
on the bar, your attention is on the lift. The Watch is right there on your wrist 
doing nothing useful. This app turns it into a hands-mostly-free coach: you select 
what to work on, click Go, lift, click Stop — and the trainer handles everything 
else (counting, progression, rest timing, motivation).

---

## What Makes This Cool

**The auto-rep counter with bounded detection.** Unlike Train Fitness or Gymatic 
which try to detect *which* exercise you're doing (hard, unreliable), this app 
sidesteps that entirely: the user selects the exercise upfront, so the Watch only 
counts motion for a *known movement pattern* in a *bounded window* (Go → Stop). 
That's the insight every competitor missed.

The motivational cues triggered by cadence drop are the second differentiator — 
the Watch is reading your fatigue in real time and responding.

Heart rate-driven rest timers (not a fixed 60s countdown) complete the picture: 
rest until you're actually recovered, not until an arbitrary timer fires.

---

## Constraints

- watchOS only (no phone companion required for core workout flow)
- Personal use first — no accounts, no cloud sync, no social layer in v1
- Apple Watch Series 4+ (accelerometer + gyroscope, HealthKit HR)
- SwiftUI + SwiftData for UI and persistence
- No OpenAI or external APIs — fully offline

---

## Premises (agreed)

1. CMMotionManager can reliably count reps for known exercises (~90%+ accuracy) 
   when exercise type is pre-selected AND bounded by Go/Stop user interaction.
2. Users accept 10-30 second setup friction (muscle group + exercise selection) 
   in exchange for hands-free counting during the set.
3. The progressive overload algorithm (add 2.5-10 lb on full completion, stop 
   adding when struggling in final reps) is the right default for v1.
4. Heart rate-driven rest timers are feasible via HealthKit real-time HR data.
5. Personal tool first — no multi-user, no cloud, no social in v1.

## Cross-Model Perspective

Independent Claude subagent reviewed the brief cold. Key findings:

**Coolest version not yet considered:** "Fatigue fingerprint" — learn your personal 
HR recovery curve per exercise over 4-6 weeks. Not just "HR < 120 = rest done" but 
YOUR specific drop rate after squats vs. curls, correlated with actual set performance. 
This is a natural v2 path from the HR rest timer.

**The technical tell:** "detects slowing/struggling and plays motivational audio cues" 
is the hardest and most exciting part. It requires real-time cadence analysis 
(rep velocity as proxy for fatigue), threshold tuning per person per exercise, and 
low-latency audio without breaking the rep-counting state machine.

**What gets you 50% there:** CMMotionManager + HealthKit (built in). The 50% to build: 
per-exercise motion templates, rep velocity calculation, the Go→Stop state machine, 
SwiftData persistence (HealthKit doesn't store weight/reps/calibration data).

**Critical data model insight:** store `peakVelocities` (seconds between rep peaks) 
from day one — it's the foundation for fatigue detection and the future 
"fatigue fingerprint" feature. The canonical data model is `ExerciseRecord` 
(SwiftData @Model, see Technical Architecture). Do not use a separate struct.

## UX Flow Design

### Key UX Decision: Go/Stop Bounded Detection

**User suggestion (incorporated):** Don't auto-detect set start/end. Instead:

1. Screen shows exercise + target weight + "GO" button
2. User taps GO → CMMotionManager activates, rep counting begins
3. Watch counts reps, announces via audio (AVSpeechSynthesizer)
4. Watch detects ~3s of no motion → shows "Stop?" prompt
5. User taps STOP to confirm (no auto-confirm — explicit control only)
6. Rest screen appears (HR + countdown timer)

**Why this is smarter:** Motion detection in a bounded Go→Stop window eliminates 
false positives (setting up the bar, transitions) and dramatically improves accuracy. 
The state machine is clean. The user has explicit control over set boundaries.

### Screen Architecture (6 screens)

**Screen 1: Muscle Picker**
- Scrollable list (Digital Crown) of muscle groups: Chest, Back, Shoulders, Biceps, 
  Triceps, Legs, Core
- Tap to toggle selection (green checkmark when selected)
- "Start →" button appears once ≥1 group selected
- Remembers last session's selection as default

**Screen 2: Exercise Queue**
- **Exercise selection algorithm:** for each selected muscle group, pick the exercise 
  with the most recent `ExerciseRecord.date` (i.e., the one the user has done most 
  recently — continue where they left off). If a muscle group has no history at all, 
  pick the first exercise in the JSON list for that group. Cap total at 5 exercises 
  (drop extras from muscle groups with the most exercises already included).
- Scrollable list (Digital Crown), each row shows: exercise name + last session weight 
  ("30 lb" or "New" for uncalibrated)
- Uncalibrated exercises show a "CAL" badge — tapping them routes to Screen 3 first
- "Begin workout" starts Screen 3 (if first exercise needs calibration) or 
  Screen 4 directly (if already calibrated)
- User can reorder or swap exercises before starting (swipe left to replace)

**Screen 3: Calibration Flow** (first time per exercise, triggered automatically 
when an uncalibrated exercise is next in the queue — user is not asked to navigate here)
- Shows exercise name + instruction: "Find weight where rep 7-8 is very hard"
- Crown/+/− to dial weight (2.5 lb increments for small muscles, 5 lb for large)
- **Bodyweight exception:** exercises with `isBodyweight: true` in the JSON (push-up, 
  pull-up, dip, plank, crunch, hanging leg raise, russian twist) skip weight dialing. 
  Show "Do 8 reps" directly. No calibrated weight stored — these exercises always 
  target the same rep count and progress via `ExerciseCalibration.calibratedWeight = 0`.
- "Start 8 reps" → immediately enters active set (Screen 4 with calibration mode)
- After set: "Was rep 8 very hard?" → Yes = calibrated / No = show adjustment 
  ("Try 5 lb more" or "Try 5 lb less") → repeat calibration at new weight

**Screen 4: Active Set (the hero screen)**
- Exercise name (small, top)
- Weight + set number (e.g., "135 lb · Set 2/3")
- Large ring progress arc filling as reps count up
- Huge rep number in center (38pt+ font, readable mid-rep)
- "GO" button (large green) before set starts
- Screen 4 always has access to `targetReps` (passed in from exercise queue). 
  Remaining reps = `targetReps - repCount`. Motivational audio fires:
  - When `remaining == 2`: "2 more, push it!"
  - When `remaining == 1`: "Last one, let's go!"
  - When fatigue flag fires mid-set (any rep): "Come on, you can do it!"
  - On set completion (repCount >= targetReps): "[X] reps. Nice work."
- Auto "Stop?" after 3s no motion, or tap STOP manually

**Screen 5: Rest Screen**
- **Elapsed timer** (top, prominent): counts up from 0:00. Always visible regardless 
  of HR data availability. Shows how long you've been resting.
- **Current heart rate** (large, red/pink): live BPM reading from HealthKit
- **Target recovery HR**: 115 BPM default (see HR adaptation spec below)
- **Progress bar**: fills as HR drops from post-set peak toward target
- **"Ready →" button**: 
  - Dimmed while HR is above target AND elapsed time < 60s
  - Goes full green when HR ≤ target (regardless of timer)
  - Also goes full green after 3 minutes elapsed regardless of HR (failsafe)
- User can tap "Ready →" at any time to override and proceed
- Shows next exercise + target weight as preview at bottom
- If HealthKit HR is unavailable (Watch not on wrist, permissions denied):
  degrade gracefully — show timer only, hide HR elements, "Ready →" 
  activates after 90s default rest

**Screen 6: Session Summary**
- Total volume (lbs × reps) for the session
- Exercises completed
- Any PRs (new weight or more reps than last session)
- "Done" → saves to SwiftData

## Progressive Overload Algorithm

```
// increment: biceps/triceps = 2.5 lb, chest/back = 5 lb, legs = 10 lb
// struggled: final 2 reps IOI > baseline IOI × 1.3 (auto-detected from peakVelocities)
// overCount: actualReps > targetReps
// weightIncreased: currentWeight > lastSessionWeight for this exercise

For each exercise after a completed set:

  // ── CASE 1: hit target, struggled ──────────────────────────────
  if actualReps >= targetReps AND struggled:
    newWeight = currentWeight          // hold — not ready to go heavier yet
    message = "Nice work."
    celebrationLevel = .standard

  // ── CASE 2: hit target, no struggle ────────────────────────────
  if actualReps >= targetReps AND NOT struggled AND NOT overCount:
    newWeight = currentWeight + increment(muscleGroup)
    message = "Solid set."
    celebrationLevel = .standard

  // ── CASE 3: over-count, same weight as last session ────────────
  // Trainee beat their target at a familiar weight — they've mastered it.
  // Treat identically to Case 2: add one increment.
  if overCount AND NOT weightIncreased:
    newWeight = currentWeight + increment(muscleGroup)
    message = "You crushed it!"
    celebrationLevel = .gold           // ring turns gold, 🔥 emoji, fanfare haptic

  // ── CASE 4: over-count, increased weight from last session ──────
  // Double win: new weight AND extra reps. The increment was too conservative.
  // Add increment again (two increments total vs. last session).
  if overCount AND weightIncreased:
    newWeight = currentWeight + increment(muscleGroup)
    // currentWeight is already +1 increment above last session,
    // so total jump = 2× increment vs. the session before
    message = "Exceptional lift!"
    celebrationLevel = .gold           // same gold treatment + "⚡ PR!" badge

  // ── CASE 5: under-count (didn't reach target) ──────────────────
  if actualReps < targetReps - 2:
    newWeight = max(currentWeight - increment(muscleGroup), minimumWeight(exercise))
    message = "Good effort. Adjusting."
    celebrationLevel = .none
    // minimumWeight: bodyweight = 0, machine = 5 lb, free weights = 2.5 lb

  // ── Close miss (targetReps - 2 <= actualReps < targetReps) ─────
  // e.g. got 7 when targeting 8 — not a deload, just hold
  if actualReps >= targetReps - 2 AND actualReps < targetReps:
    newWeight = currentWeight          // hold — try again at same weight
    message = "Almost there."
    celebrationLevel = .none

targetReps = 8 (default, fixed for v1)
```

"Struggled" and "overCount" are both detected automatically from `repIntervals` — 
no user input required. `celebrationLevel` drives the visual and haptic treatment 
on Screen 4c:
- `.none` — plain white number, neutral message
- `.standard` — white number, green "Rest →" button  
- `.gold` — yellow number, full gold ring, strong haptic, animated emoji badge

## Exercise Library (Local Database)

Store as a JSON bundle in the app. Source: Healthline muscle group guide.
Structure:
```json
{
  "chest": ["bench_press", "incline_press", "chest_fly", "push_up", "cable_crossover"],
  "back": ["pull_up", "bent_over_row", "lat_pulldown", "seated_row", "deadlift"],
  "shoulders": ["overhead_press", "lateral_raise", "front_raise", "face_pull"],
  "biceps": ["bicep_curl", "hammer_curl", "concentration_curl"],
  "triceps": ["tricep_pushdown", "skull_crusher", "overhead_extension", "dip"],
  "legs": ["squat", "leg_press", "lunge", "leg_extension", "leg_curl", "calf_raise"],
  "core": ["plank", "crunch", "hanging_leg_raise", "russian_twist"]
}
```

For each exercise, store:
```json
{
  "id": "bench_press",
  "name": "Bench Press",
  "muscleGroups": ["chest", "triceps"],
  "primaryAxis": "z",
  "defaultThreshold": 0.4,
  "increment": { "small": 2.5, "large": 5.0 },
  "incrementGroup": "large",
  "isBodyweight": false,
  "minimumWeight": 2.5,
  "defaultStartingWeight": 45.0
}
```

Digital Crown scrolling: use `.focusable()` + `.digitalCrownRotation($value)` 
on the weight picker. Bind to a `Double` and snap to nearest `increment` step.

## Technical Architecture

### Stack
- SwiftUI (watchOS native UI)
- SwiftData (local persistence — sets, calibration, progress)
- CMMotionManager (accelerometer + gyroscope at 60Hz)
- HealthKit (real-time heart rate via HKAnchoredObjectQuery)
- AVSpeechSynthesizer (audio coaching)
- WatchKit (haptic feedback: WKInterfaceDevice.current().play(.success))

### Rep Counting — Signal Processing

**Algorithm: rolling-max peak detection with debounce**

```swift
CMMotionManager.startAccelerometerUpdates(to: .main) { data, _ in
    let sample = data.acceleration[primaryAxis]  // x, y, or z per exercise JSON
    sampleBuffer.append(sample)
    if sampleBuffer.count > 30 { sampleBuffer.removeFirst() }  // 30-sample / 500ms window

    // Peak detection: a rep is counted when:
    //   1. sample > detectionThreshold (from ExerciseCalibration, default 0.4g)
    //   2. sample == max(sampleBuffer)  (it's the current rolling max — first-deriv zero-crossing)
    //   3. Date.now - lastRepTime > 0.4s  (400ms debounce — prevents double-count)
    if sample > threshold && sample == sampleBuffer.max()! &&
       Date().timeIntervalSince(lastRepTime) > 0.4 {
        repCount += 1
        lastRepTime = Date()
        repIntervals.append(Date().timeIntervalSince(prevRepTime))  // IOI in seconds
        prevRepTime = Date()
        announceRep(repCount, targetReps: targetReps)
    }
}
```

**Fatigue detection (rep interval analysis):**
- Baseline IOI: average of intervals for reps 2, 3, 4 (skip rep 1 — startup is noisy)
- If set has fewer than 4 reps, skip fatigue detection for that set
- Fatigue flag: if current IOI > baseline IOI × 1.3 (i.e., 30% slower)
- When fatigue flag fires: trigger motivational audio cue
- `struggled` (stored in ExerciseRecord) = true if final 2 reps both flagged as fatigued

**Note on naming:** `ExerciseRecord.peakVelocities` stores rep inter-onset intervals 
(seconds between rep peaks) — lower value = faster rep. "Velocity" here means 
cadence speed, not physical velocity. The field name is kept for future-proofing 
(v2 may compute physical velocity from raw acceleration).

**Error handling:** if `CMMotionManager.isAccelerometerAvailable == false` or 
authorization denied → degrade to manual mode: show large +/− buttons on Screen 4, 
disable auto-count, log the reason in `ExerciseRecord.manualOverride = true`.

Default threshold = 0.4g if no `ExerciseCalibration` exists yet.
`detectionThreshold` updates silently after calibration set completes 
(before the "Was rep 8 very hard?" confirmation): 
`newThreshold = 0.8 × median(peakAmplitudes observed during calibration set)`

### Data Model (SwiftData)
```swift
@Model class ExerciseRecord {
    var exerciseId: String
    var date: Date
    var targetWeight: Double
    var targetReps: Int
    var actualReps: Int
    var repIntervals: [Double]     // inter-onset intervals in seconds (lower = faster rep)
    var struggled: Bool            // true if final 2 reps IOI > baseline IOI × 1.3
    var manualOverride: Bool       // true if rep counter was in manual mode (no CMMotionManager)
    // Note: calibration state lives in ExerciseCalibration only.
    // An exercise is calibrated iff ExerciseCalibration record exists for its exerciseId.
}

@Model class RestRecord {
    var exerciseId: String
    var date: Date
    var restDurationSeconds: Double
    var startHR: Double
    var endHR: Double
    var userOverrode: Bool         // tapped Ready before HR target reached
}

@Model class ExerciseCalibration {
    var exerciseId: String
    var calibratedWeight: Double
    var calibrationDate: Date
    var detectionThreshold: Double  // tunable per exercise
}
```

### HKWorkoutSession (required for HR access)

HealthKit real-time HR during exercise requires an active `HKWorkoutSession`. 
This is non-trivial and must be set up before the first set begins.

```swift
// Start at workout begin (not per-set)
let config = HKWorkoutConfiguration()
config.activityType = .traditionalStrengthTraining
config.locationType = .indoor

let session = try HKWorkoutSession(healthStore: healthStore, configuration: config)
let builder = session.associatedWorkoutBuilder()
builder.dataSource = HKLiveWorkoutDataSource(healthStore: healthStore,
                                              workoutConfiguration: config)
session.startActivity(with: Date())
builder.beginCollection(withStart: Date()) { _, _ in }
```

Without an active session, `HKAnchoredObjectQuery` will not return live HR updates 
on watchOS. Session must be ended and saved when workout completes: call 
`session.end()` + `builder.endCollection(withEnd: Date())` + `builder.finishWorkout()` 
on the Session Summary "Done" button tap. On app crash or Watch removal, 
`HKWorkoutSession` ends automatically — partial workout data is lost. Acceptable 
for v1. Add per-set SwiftData auto-save for crash recovery in v2.

### Heart Rate Rest Timers
- Subscribe to live HR via `HKLiveWorkoutDataSource` (within the workout session above)
- **Recovery target HR:** starts at 115 BPM default
- **HR adaptation (simple, v1):** fixed at 115 BPM for v1. No adaptive logic in v1.
  Log `(restDuration, endHR, startHR, userOverrode: Bool)` per rest period for 
  future v2 adaptation. Manual override in Settings ("Rest HR target: 115 BPM" 
  — user can change this themselves).
- "Ready" fires when current HR ≤ target for 10+ consecutive seconds
- Elapsed timer always runs independently from HR logic (see Screen 5 spec)
- v2: per-exercise recovery curves, fatigue fingerprint

## Recommended Approach

**Approach B**, executed in this order:

**Weekend 1:** CMMotionManager rep counter for bench press only. One screen.
Get it accurate enough to trust (>85% correct on a set of 8). Nothing else.

**Weekend 2:** SwiftData model, calibration flow, exercise library JSON, muscle 
picker. All screens wired. Still bench press only.

**Weekend 3:** Expand to 5 exercises across 3 muscle groups. Tune thresholds 
per exercise. Add progressive overload algorithm. (Remaining 21+ exercises 
can be added incrementally as you actually use them — add each one to the JSON 
and test at the gym. Not a blocker for usability.)

**Weekend 4:** HealthKit HR integration + rest timers. Audio coaching 
(AVSpeechSynthesizer). Session summary screen.

**Weekend 5:** Polish. Rep velocity-based fatigue detection. Calibration 
refinement. Use it for real.

## Open Questions

1. **Rep detection per-exercise tuning:** Deadlift and squat have ambiguous motion 
   axes (Z-axis for standing hip hinge vs. Y-axis for squat depth). Start with 
   bench press and bicep curl (clear vertical motion), add lower-body exercises 
   last. Expect 1-2 gym sessions of threshold tuning per exercise.

2. **watchOS memory/CPU limits:** CMMotionManager at 60Hz + HKWorkoutSession + 
   AVSpeechSynthesizer on Series 4 is a real risk. First contingency: drop to 30Hz. 
   Second contingency: pre-record the 6 audio cues as MP3 files, play via 
   AVAudioPlayer (avoids on-device synthesis overhead). Test on real hardware 
   in Weekend 1 before assuming it's fine.

3. **AVSpeechSynthesizer latency:** Measure actual latency in Weekend 1 by logging 
   `Date()` at fatigue-flag-fire vs. audio-start. If > 500ms, switch to 
   pre-recorded MP3s. Pre-record: "1", "2" ... "8", "2 more push it", "last one 
   let's go", "come on you can do it", "nice work".
