## Problem Statement

Every gym app requires you to tap, log, or interact between sets. Your hands are 
on the bar, your attention is on the lift. The Watch is right there on your wrist 
doing nothing useful. This app turns it into a hands-mostly-free coach: you select 
what to work on, click Go, lift, click Stop — and the trainer handles everything 
else (counting, progression, rest timing, motivation).

---

## What Makes This Cool

**The auto-rep counter with bounded detection.** Unlike Train Fitness or Gymatic 
which try to detect *which* exercise you're doing (hard, unreliable), this app 
sidesteps that entirely: the user selects the exercise upfront, so the Watch only 
counts motion for a *known movement pattern* in a *bounded window* (Go → Stop). 
That's the insight every competitor missed.

The motivational cues triggered by cadence drop are the second differentiator — 
the Watch is reading your fatigue in real time and responding.

Heart rate-driven rest timers (not a fixed 60s countdown) complete the picture: 
rest until you're actually recovered, not until an arbitrary timer fires.

## Constraints

- watchOS only (no phone companion required for core workout flow)
- Personal use first — no accounts, no cloud sync, no social layer in v1
- Apple Watch Series 4+ (accelerometer + gyroscope, HealthKit HR)
- SwiftUI + SwiftData for UI and persistence
- No OpenAI or external APIs — fully offline

---

## Premises (agreed)

1. CMMotionManager can reliably count reps for known exercises (~90%+ accuracy) 
   when exercise type is pre-selected AND bounded by Go/Stop user interaction.
2. Users accept 10-30 second setup friction (muscle group + exercise selection) 
   in exchange for hands-free counting during the set.
3. The progressive overload algorithm (add 2.5-10 lb on full completion, stop 
   adding when struggling in final reps) is the right default for v1.
4. Heart rate-driven rest timers are feasible via HealthKit real-time HR data.
5. Personal tool first — no multi-user, no cloud, no social in v1.

## Cross-Model Perspective

Independent Claude subagent reviewed the brief cold. Key findings:

**Coolest version not yet considered:** "Fatigue fingerprint" — learn your personal 
HR recovery curve per exercise over 4-6 weeks. Not just "HR < 120 = rest done" but 
YOUR specific drop rate after squats vs. curls, correlated with actual set performance. 
This is a natural v2 path from the HR rest timer.

**The technical tell:** "detects slowing/struggling and plays motivational audio cues" 
is the hardest and most exciting part. It requires real-time cadence analysis 
(rep velocity as proxy for fatigue), threshold tuning per person per exercise, and 
low-latency audio without breaking the rep-counting state machine.

**What gets you 50% there:** CMMotionManager + HealthKit (built in). The 50% to build: 
per-exercise motion templates, rep velocity calculation, the Go→Stop state machine, 
SwiftData persistence (HealthKit doesn't store weight/reps/calibration data).

**Critical data model insight:** store `peakVelocities` (seconds between rep peaks) 
from day one — it's the foundation for fatigue detection and the future 
"fatigue fingerprint" feature. The canonical data model is `ExerciseRecord` 
(SwiftData @Model, see Technical Architecture). Do not use a separate struct.

## UX Flow Design

### Key UX Decision: Go/Stop Bounded Detection

**User suggestion (incorporated):** Don't auto-detect set start/end. Instead:

1. Screen shows exercise + target weight + "GO" button
2. User taps GO → CMMotionManager activates, rep counting begins
3. Watch counts reps, announces via audio (AVSpeechSynthesizer)
4. Watch detects ~3s of no motion → shows "Stop?" prompt
5. User taps STOP to confirm (no auto-confirm — explicit control only)
6. Rest screen appears (HR + countdown timer)

**Why this is smarter:** Motion detection in a bounded Go→Stop window eliminates 
false positives (setting up the bar, transitions) and dramatically improves accuracy. 
The state machine is clean. The user has explicit control over set boundaries.

### Screen Architecture (6 screens)

**Screen 1: Muscle Picker**
- **Primary**: Scrollable list of muscle groups (Chest, Back, Shoulders, Biceps, Triceps, Legs, Core)
- **Tertiary**: "Start →" button (bottom, disabled until ≥1 selected)
- Digital Crown OR tap-to-select (sweaty/chalked hands fallback)
- Tap to toggle selection (green checkmark when selected)
- Remembers last session's selection as default
- Visual flow: List fills screen, button sits at bottom. When scrolling, button stays visible at screen bottom.

**Screen 2: Exercise Queue**
- **Primary**: List of exercises in workout order (first exercise = current, highlighted with green border)
- **Secondary**: Exercise name + last session weight ("30 lb" or "New" for uncalibrated)
- **Tertiary**: "CAL" badge for uncalibrated exercises (small, right-aligned)
- **Exercise selection algorithm:** for each selected muscle group, pick the exercise 
  with the most recent `ExerciseRecord.date` (i.e., the one the user has done most 
  recently — continue where they left off). If a muscle group has no history at all, 
  pick the first exercise in the JSON list for that group. Cap total at 5 exercises 
  (drop extras from muscle groups with the most exercises already included).
- Scrollable list (Digital Crown), each row shows: exercise name + last session weight 
  ("30 lb" or "New" for uncalibrated)
- Uncalibrated exercises show a "CAL" badge — tapping them routes to Screen 3 first
- Current exercise has green left border (2pt), bold name
- "Begin workout" starts Screen 3 (if first exercise needs calibration) or 
  Screen 4 directly (if already calibrated)
- User can reorder or swap exercises before starting (swipe left to replace)

**Screen 3: Calibration Flow** (first time per exercise, triggered automatically 
when an uncalibrated exercise is next in the queue — user is not asked to navigate here)
- Shows exercise name + instruction: "Find weight where rep 7-8 is very hard"
- Digital Crown OR explicit +/− buttons to dial weight (2.5 lb increments for small muscles, 5 lb for large). Both input paths are always visible — Crown is not the only option.
- **Bodyweight exception:** exercises with `isBodyweight: true` in the JSON (push-up, 
  pull-up, dip, plank, crunch, hanging leg raise, russian twist) skip weight dialing. 
  Show "Do 8 reps" (or "Hold for 30s" for isometric) directly. No calibrated weight stored — bodyweight exercises use the JSON `defaultThreshold` forever, never create an `ExerciseCalibration` record.
- "Start 8 reps" → immediately enters active set (Screen 4 with calibration mode)
- After set: "Was rep 8 very hard?" → Yes = calibrated / No = show adjustment 
  ("Try 5 lb more" or "Try 5 lb less") → repeat calibration at new weight
- **Retry cap:** after 3 consecutive "No, too easy" taps, escalate to a manual weight-entry screen. Persist `ExerciseCalibration` with the last attempted weight, `detectionThreshold` from the latest set, and `manualEntry = true`. Prevents infinite loop.

**Screen 4: Active Set (the hero screen)**
- **Primary**: Huge rep number / time in center (42pt Söhne Bold, readable mid-rep, Always-On display dim-safe) — white text on dark background for contrast
- **Secondary**: Large ring progress arc filling as reps count up (or as time elapses for isometric)
- **Tertiary**: Exercise name (small, top, 12pt), Weight + set number (e.g., "135 lb · Set 2/3" in 14pt)
- "GO" button (large green) before set starts
- Rep count is wrapped in an `@Observable` class to scope re-renders — only the number and ring view observe it.
- **Rep-counted mode** (non-isometric):
  - `targetReps` passed in from exercise queue. Remaining = `targetReps - repCount`.
  - Motivational audio fires:
    - `remaining == 2`: "2 more, push it!"
    - `remaining == 1`: "Last one, let's go!"
    - Fatigue flag transition mid-set: "Come on, you can do it!"
    - `repCount >= targetReps`: "[X] reps. Nice work."
  - Auto "Stop?" prompt requires **both** 8 seconds of no motion AND a rising-edge motion signal since the last detected rep (prevents false trigger on deadlift lockout / squat pause). Manual STOP always available.
- **Isometric mode** (when `exercise.isIsometric == true`):
  - Count-up timer instead of rep number. Target = N seconds (e.g., 30s plank).
  - Rep detector is NOT running — no auto "Stop?" prompt.
  - User taps STOP manually when done. Audio cue at target: "Nice hold."
  - `ExerciseRecord.actualReps` stores elapsed seconds.
- Visual flow: Rep number dominates screen. Ring wraps rep number. Exercise name at top-left, weight info at top-right.

**Screen 5: Rest Screen**
- **Primary**: Current heart rate (large, red/pink, 56pt Söhne Bold) — live BPM reading from HealthKit
- **Secondary**: "Ready →" button (large, green) — status indicated by opacity/colored border
- **Tertiary**: Progress bar (HR drop from peak to target), Elapsed timer (top, 14pt, gray)
- **Target recovery HR**: 115 BPM default (see HR adaptation spec below)
- **"Ready →" button**: 
  - Dimmed while HR is above target AND elapsed time < 60s
  - Goes full green when HR ≤ target (regardless of timer)
  - Also goes full green after 3 minutes elapsed regardless of HR (failsafe)
- User can tap "Ready →" at any time to override and proceed
- Shows next exercise + target weight as preview at bottom
- If HealthKit HR is unavailable (Watch not on wrist, permissions denied):
  degrade gracefully — show timer only, hide HR elements, "Ready →" 
  activates after 90s default rest
- Visual flow: HR number dominates (red is attention-grabbing). Progress bar provides context. Timer at top for elapsed tracking. Ready button is the clear action.

**Screen 6: Session Summary**
- **Primary**: Total volume (lbs × reps) in large center text (42pt Söhne Bold)
- **Secondary**: PR badge (gold star icon) if any new records
- **Tertiary**: List of exercises completed (small list, 12pt)
- "Done" → saves to SwiftData
- Visual flow: Volume number is the celebrate moment. PR badges provide social validation. Exercise list is for reference.

## Progressive Overload Algorithm

```
// increment: biceps/triceps = 2.5 lb, chest/back = 5 lb, legs = 10 lb
// struggled: final 2 reps IOI > baseline IOI × 1.3 (auto-detected from peakVelocities)
// overCount: actualReps > targetReps
// weightIncreased: currentWeight > lastSessionWeight for this exercise

For each exercise after a completed set:

  // ── CASE 1: hit target, struggled ──────────────────────────────
  if actualReps >= targetReps AND struggled:
    newWeight = currentWeight          // hold — not ready to go heavier yet
    message = "Nice work."
    celebrationLevel = .standard

  // ── CASE 2: hit target, no struggle ────────────────────────────
  if actualReps >= targetReps AND NOT struggled AND NOT overCount:
    newWeight = currentWeight + increment(muscleGroup)
    message = "Solid set."
    celebrationLevel = .standard

  // ── CASE 3: over-count, same weight as last session ────────────
  // Trainee beat their target at a familiar weight — they've mastered it.
  // Treat identically to Case 2: add one increment.
  if overCount AND NOT weightIncreased:
    newWeight = currentWeight + increment(muscleGroup)
    message = "You crushed it!"
    celebrationLevel = .gold           // ring turns gold, 🔥 emoji, fanfare haptic

  // ── CASE 4: over-count, increased weight from last session ──────
  // Double win: new weight AND extra reps. The increment was too conservative.
  // Add increment again (two increments total vs. last session).
  if overCount AND weightIncreased:
    newWeight = currentWeight + increment(muscleGroup)
    // currentWeight is already +1 increment above last session,
    // so total jump = 2× increment vs. the session before
    message = "Exceptional lift!"
    celebrationLevel = .gold           // same gold treatment + "⚡ PR!" badge

  // ── CASE 5: under-count (didn't reach target) ──────────────────
  if actualReps < targetReps - 2:
    newWeight = max(currentWeight - increment(muscleGroup), minimumWeight(exercise))
    message = "Good effort. Adjusting."
    celebrationLevel = .none
    // minimumWeight: bodyweight = 0, machine = 5 lb, free weights = 2.5 lb

  // ── Close miss (targetReps - 2 <= actualReps < targetReps) ─────
  // e.g. got 7 when targeting 8 — not a deload, just hold
  if actualReps >= targetReps - 2 AND actualReps < targetReps:
    newWeight = currentWeight          // hold — try again at same weight
    message = "Almost there."
    celebrationLevel = .none

targetReps = 8 (default, fixed for v1)
```

"Struggled" and "overCount" are both detected automatically from `repIntervals` — 
no user input required. `celebrationLevel` drives the visual and haptic treatment 
on Screen 4c:
- `.none` — plain white number, neutral message
- `.standard` — white number, green "Rest →" button  
- `.gold` — yellow number, full gold ring, strong haptic, animated emoji badge

**Implementation note:** the 5 cases above live in a pure function `ProgressiveOverload.compute(input: OverloadInput) -> OverloadResult` — not scattered across a view controller. `OverloadInput` bundles `(currentWeight, lastSessionWeight, actualReps, targetReps, struggled, muscleGroup, weightType)`. `OverloadResult` bundles `(newWeight, message, celebrationLevel)`. Fully testable in isolation, no SwiftData or UI dependencies. See Test Review → Swift Testing suite.

**Case 4 math clarification:** `currentWeight` on entry to Case 4 is already `lastSessionWeight + increment` (from an earlier session's Case 2 or 3). Adding another `increment` in Case 4 yields a total jump of 2 × `increment` from the user's *last recorded session*, even though the code only adds one increment in this step. The "Exceptional lift!" message reflects the cumulative double-up.

**Under-count floor:** `minimumWeight` is derived from `exercise.weightType`: `free = 2.5 lb`, `machine = 5 lb`, `cable = 5 lb`, `bodyweight = 0` (bodyweight exercises never change `calibratedWeight`; progressive overload is a no-op for them — they always target the same rep count).

  "legs": ["squat", "leg_press", "lunge", "leg_extension", "leg_curl", "calf_raise"],
  "core": ["plank", "crunch", "hanging_leg_raise", "russian_twist"]

For each exercise, store:
```json
{
  "id": "bench_press",
  "name": "Bench Press",
  "muscleGroups": ["chest", "triceps"],
  "defaultThreshold": 0.4,
  "increment": { "small": 2.5, "large": 5.0 },
  "incrementGroup": "large",

```

**Schema field reference:**
- `defaultThreshold` (double, g-force): fallback detection threshold when no `ExerciseCalibration` exists. Bodyweight exercises always use this — they never calibrate.
- `isBodyweight` (bool): skips the weight-dial step of calibration, stores `calibratedWeight = 0`.
- `isIsometric` (bool): `true` for plank, crunch, hanging leg raise, russian twist. Isometric exercises use a time-held counter instead of a rep counter; the 3s-no-motion "Stop?" auto-prompt is disabled for them (user must tap STOP manually).
- `weightType` ("free" | "machine" | "cable" | "bodyweight"): drives the `minimumWeight` floor (free = 2.5 lb, machine = 5 lb, cable = 5 lb, bodyweight = 0).
- **`primaryAxis` was removed.** The rep detector is orientation-independent now — see Technical Architecture → Rep Counting.

Digital Crown scrolling: use `.focusable()` + `.digitalCrownRotation($value)` 
on the weight picker. Bind to a `Double` and snap to nearest `increment` step.

## Technical Architecture

### Stack
- SwiftUI (watchOS native UI)
- SwiftData (local persistence — sets, calibration, progress)
- CMMotionManager (accelerometer + gyroscope at 60Hz)
- HealthKit (real-time heart rate via HKAnchoredObjectQuery)
- AVSpeechSynthesizer (audio coaching)
- WatchKit (haptic feedback: WKInterfaceDevice.current().play(.success))

### Rep Counting — Signal Processing

**Algorithm: center-of-window local-max peak detection on orientation-independent acceleration magnitude, with debounce.**

Key design decisions (from /plan-eng-review):
1. Use `CMDeviceMotion.userAcceleration` (gravity already removed by the OS) instead of raw `CMAccelerometerData`. The coordinate frame is still device-local, but we take the **magnitude** `sqrt(x² + y² + z²)` which is invariant under rotation — works regardless of which wrist, band orientation, or facing direction.
2. Detect peaks at the **center of a 30-sample sliding window** rather than the newest sample. The lagging-max approach fires on any rising edge that exceeds the recent max, missing the true local peak. Center-of-window means a true local maximum, with a ~250ms lag (acceptable for rep counting).
3. Run callbacks on a **dedicated `OperationQueue`**, not `.main`. Prevents contention with SwiftUI rendering and audio at 30Hz+.
4. Use **monotonic clock** (`CFAbsoluteTimeGetCurrent()`) for rep interval timing, not `Date()`. NTP adjustments can shift wall-clock time mid-set.

```swift
// MotionSource protocol (injected — see DI section)
// Rep detector is fronted by protocols so it is testable against recorded fixtures.

final class RepDetector {
    private let threshold: Double              // from ExerciseCalibration, else JSON defaultThreshold
    private let debounceSeconds: Double = 0.4  // 400ms minimum between reps
    private let bufferSize: Int = 30           // 30 samples @ 30Hz = 1s window
    private let centerIdx: Int = 15            // index N/2

    private var sampleBuffer: [(t: CFAbsoluteTime, mag: Double)] = []
    private var lastRepTime: CFAbsoluteTime = 0
    private(set) var repCount: Int = 0
    private(set) var repIntervals: [Double] = []  // inter-onset intervals, seconds

    /// Called on the motion queue once per sample.
    func processSample(t: CFAbsoluteTime, userAcceleration: SIMD3<Double>) {
        let mag = sqrt(userAcceleration.x * userAcceleration.x
                     + userAcceleration.y * userAcceleration.y
                     + userAcceleration.z * userAcceleration.z)
        sampleBuffer.append((t, mag))
        if sampleBuffer.count > bufferSize { sampleBuffer.removeFirst() }
        guard sampleBuffer.count == bufferSize else { return }

        // Center-of-window local max: the sample at index centerIdx must be the max
        // of the full 30-sample buffer AND above threshold AND past debounce.
        let centerSample = sampleBuffer[centerIdx]
        guard centerSample.mag > threshold else { return }
        let windowMax = sampleBuffer.map { $0.mag }.max() ?? 0
        guard centerSample.mag >= windowMax else { return }
        guard centerSample.t - lastRepTime > debounceSeconds else { return }

        // Count the rep.
        repCount += 1
        if lastRepTime > 0 {
            repIntervals.append(centerSample.t - lastRepTime)
        }
        lastRepTime = centerSample.t
        // announcer / haptic engine fire via injected protocols
    }

    func reset() {
        sampleBuffer.removeAll(keepingCapacity: true)
        lastRepTime = 0
        repCount = 0
        repIntervals.removeAll(keepingCapacity: true)
    }
}
```

**Threshold resolution:** `threshold = ExerciseCalibration.detectionThreshold ?? exercise.defaultThreshold` at RepDetector init time. Bodyweight exercises never have a calibration record, so they always use `exercise.defaultThreshold` from the JSON (0.4 by default).

**Sample rate:** 30Hz default (`motionUpdateInterval = 1.0 / 30.0`). Gives 33ms samples, 1s buffer window, 500ms half-window around center. Bump to 60Hz only if fidelity is poor on real hardware — measure in Weekend 1.

**Fatigue detection (rep interval analysis):**
- Baseline IOI: average of intervals for reps 2, 3, 4 (skip rep 1 — startup is noisy, and `repIntervals[0]` is only recorded on rep 2 since rep 1 has no prior)
- If set has fewer than 5 reps, skip fatigue detection entirely (no `struggled` flag — weight holds for progressive overload)
- Fatigue flag: if current IOI > baseline IOI × 1.3 (i.e., 30% slower)
- When fatigue flag transitions from false → true mid-set: fire motivational audio cue via `SpeechAnnouncer` protocol
- `struggled` (stored in ExerciseRecord) = true if final 2 reps both flagged as fatigued

**Isometric exercises:** when `exercise.isIsometric == true`, the rep detector is not used. Screen 4 shows a count-up timer ("0:23") instead of a rep counter. The 3s-no-motion auto-prompt is disabled. User taps STOP manually. ExerciseRecord stores `actualReps = elapsedSeconds` with a sentinel flag or separate field (TBD at implementation time).

**Non-isometric lockout handling:** deadlift and squat pauses at the top/bottom of a rep can exceed 3s of near-zero motion. To prevent false "Stop?" prompts:
- Extend the no-motion threshold from 3s → 8s
- AND require a rising-edge motion signal (mag > threshold at any point) since the last detected rep before the prompt fires
- If neither condition is met (true rest), show the prompt

**Note on naming:** `ExerciseRecord.repIntervals` (renamed from `peakVelocities` for clarity) stores inter-onset intervals in seconds — lower value = faster rep. "Velocity" would mean physical velocity (m/s), which is a v2 feature. See TODOS.md.

**Error handling:**
- `CMMotionManager.isDeviceMotionAvailable == false` or authorization denied → degrade to manual mode: show large +/− buttons on Screen 4, disable auto-count, log `ExerciseRecord.manualOverride = true`
- `HKWorkoutSession` start failure → show "Heart rate unavailable. Continue without HR rest timers?" prompt. If accepted, workout continues in degraded mode: rep detection still works, rest screen uses 90s fixed timer, no HR UI.

**Interaction State Coverage:**
| Screen | Feature | Loading | Empty | Error | Success | Partial |
|--------|---------|---------|-------|-------|---------|---------|

**Calibration threshold tuning:** after the calibration set completes (before the "Was rep 8 very hard?" confirmation), `detectionThreshold` is updated silently: `newThreshold = 0.8 × median(magnitudes observed at each detected rep during calibration)`.

**Calibration retry cap:** if the user taps "No, too easy" 3 times in a row, the flow escalates to a manual weight-entry screen and persists `ExerciseCalibration` with the last attempted weight and `detectionThreshold`. Prevents an infinite loop.

### Audio Session Configuration

Before starting `AVSpeechSynthesizer`, configure `AVAudioSession` explicitly. Without this, TTS may not route to the watch speaker during an active `HKWorkoutSession(.traditionalStrengthTraining)`.

```swift
let audio = AVAudioSession.sharedInstance()
try audio.setCategory(.playback, mode: .default, options: [.duckOthers, .mixWithOthers])
try audio.setActive(true, options: [.notifyOthersOnDeactivation])
```

**Weekend 1 gate:** before writing any rep-detector code, run a manual on-device spike: start an `HKWorkoutSession`, configure `AVAudioSession` as above, speak a test phrase via `AVSpeechSynthesizer`, confirm it comes out of the watch speaker. If it does not, AirPods become a documented requirement and Screen 4 adds an AirPods-status icon. **Do not assume this works.**

### Backgrounding — HKWorkoutSession from day 1

`HKWorkoutSession` is started at "Begin workout" (end of Screen 2), not at the first set. This gives the app:
- Free background execution while the wrist is down mid-set
- Free live HR via `HKLiveWorkoutDataSource`
- Free audio routing during the workout

The session is ended + `builder.finishWorkout()` is called at the Session Summary "Done" tap. On crash or Watch removal, the session ends automatically — partial workout data is lost for v1 (accepted). See TODOS.md for per-set auto-save.

### Data Model (SwiftData)

**Schema versioning (v1):** add a `ModelSchemaVersion` const in code. At app launch, compare it to a stored value in `UserDefaults`. If they differ, wipe the `ModelContainer` and recreate. Personal-use v1 accepts history loss on schema change. Proper migrations are v2.

```swift
let ModelSchemaVersion = 1  // bump this when @Model classes change

@Model class ExerciseRecord {
    var exerciseId: String
    var date: Date
    var targetWeight: Double
    var targetReps: Int
    var actualReps: Int
    var repIntervals: [Double]     // inter-onset intervals in seconds (lower = faster rep)
    var struggled: Bool            // true if final 2 reps IOI > baseline IOI × 1.3
    var manualOverride: Bool       // true if rep counter was in manual mode (no CMDeviceMotion)
    var degradedHR: Bool           // true if HKWorkoutSession unavailable, workout ran without HR
    // Note: calibration state lives in ExerciseCalibration only.
    // An exercise is calibrated iff an ExerciseCalibration record exists for its exerciseId.
    // Bodyweight exercises never get an ExerciseCalibration — they use exercise.defaultThreshold.
}

@Model class RestRecord {
    var exerciseId: String
    var date: Date
    var restDurationSeconds: Double
    var startHR: Double            // 0 if degraded mode (HR unavailable)
    var endHR: Double              // 0 if degraded mode
    var userOverrode: Bool         // tapped Ready before HR target reached
}

@Model class ExerciseCalibration {
    var exerciseId: String
    var calibratedWeight: Double
    var calibrationDate: Date
    var detectionThreshold: Double // tunable per exercise, updated after each calibration set
    var manualEntry: Bool          // true if user hit the 3-retry cap and entered weight manually
}
```

### HKWorkoutSession (required for HR access AND backgrounding)

HealthKit real-time HR during exercise requires an active `HKWorkoutSession`. **It is also the mechanism that keeps the app alive in the background when the wrist drops mid-set.** For both reasons, the session starts at "Begin workout" (end of Screen 2), not per-set or per-exercise.

```swift
// Start at workout begin (not per-set)
let config = HKWorkoutConfiguration()
config.activityType = .traditionalStrengthTraining
config.locationType = .indoor

let session = try HKWorkoutSession(healthStore: healthStore, configuration: config)
let builder = session.associatedWorkoutBuilder()
builder.dataSource = HKLiveWorkoutDataSource(healthStore: healthStore,
                                              workoutConfiguration: config)
session.startActivity(with: Date())
builder.beginCollection(withStart: Date()) { _, _ in }
```

Without an active session, `HKAnchoredObjectQuery` will not return live HR updates 
on watchOS AND the app will be suspended the moment the wrist drops. Session must be ended and saved when workout completes: call 
`session.end()` + `builder.endCollection(withEnd: Date())` + `builder.finishWorkout()` 
on the Session Summary "Done" button tap. On app crash or Watch removal, 
`HKWorkoutSession` ends automatically — partial workout data is lost. Acceptable 
for v1. See TODOS.md for per-set auto-save in v2.

**Session start-failure path:** if `HKWorkoutSession` fails to start (permission denied, HealthKit disabled, system busy), show a clear error screen: "Heart rate unavailable. Continue without HR rest timers?" If the user accepts, the workout proceeds in **degraded mode** — rep detection still works via `CMMotionManager`, rest screen uses a fixed 90s timer, no HR UI, `ExerciseRecord.degradedHR = true`. Do NOT crash or silently fail.

### Heart Rate Rest Timers
- Subscribe to live HR via `HKLiveWorkoutDataSource` (within the workout session above)
- **Recovery target HR:** starts at 115 BPM default
- **HR adaptation (simple, v1):** fixed at 115 BPM for v1. No adaptive logic in v1.
  Log `(restDuration, endHR, startHR, userOverrode: Bool)` per rest period for 
  future v2 adaptation. Manual override in Settings ("Rest HR target: 115 BPM" 
  — user can change this themselves).
- "Ready" fires when current HR ≤ target for 10+ consecutive seconds
- Elapsed timer always runs independently from HR logic (see Screen 5 spec)
- v2: per-exercise recovery curves, fatigue fingerprint
